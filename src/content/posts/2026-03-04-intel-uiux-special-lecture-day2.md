3월 4일 UI/UX 특강은 전날보다 훨씬 더 실무 프로세스 쪽에 가까웠다. 3월 3일이 "좋은 UX란 무엇인가"를 이해하는 날이었다면, 이번 강의는 "그 좋은 UX를 실제로 어떻게 만들어 가는가"를 배우는 시간이었다. 화면을 바로 그리는 것이 아니라 문제를 발견하고, 정의하고, 아이디어를 발전시키고, 검증하고, 운영 단계까지 연결하는 흐름 전체를 다뤘다.

강의의 중심에는 더블 다이아몬드 프로세스가 있었다. `Discover -> Define -> Develop -> Deliver` 네 단계로 나뉘고, 앞에서는 문제를 넓게 탐색하고 뒤에서는 해결안을 넓게 만든 다음 다시 좁혀서 실행하는 구조다. 예전에는 디자인이 어느 순간 갑자기 예쁜 시안으로 나타나는 일처럼 느껴졌는데, 이번 특강을 들으면서 디자인도 상당히 구조적인 문제 해결 과정이라는 점이 더 분명해졌다.

## 더블 다이아몬드로 보는 UI/UX 디자인 흐름

더블 다이아몬드는 해결할 문제를 찾는 구간과, 발견한 문제를 해결하는 구간으로 나뉜다. 문제를 처음부터 정답처럼 고정하지 않고 먼저 충분히 조사하고 확산한 뒤, 핵심 문제를 정의하고, 그다음 해결안을 여러 방향으로 탐색해서 다시 검증하고 전달하는 구조다. 이 모델이 좋았던 이유는, 성급하게 솔루션부터 만드는 습관을 경계하게 해주기 때문이다.

![문제 탐색과 해결을 나눈 더블 다이아몬드](/images/posts/intel-uiux-special-lecture-day2/image3.jpg)

정의하기 단계에서는 HMW 문제 정의, 사용자 니즈 정리, 리서치 인사이트 정리, 페르소나 설정, 공감 지도, 사용자 정의, 우선순위 설정, 사용자 여정지도, 시나리오 정의, 서비스 블루프린트, KPI 설정까지 이어진다고 설명했다. 단순히 조사한 사실을 나열하는 것이 아니라, 그 사실을 구조화해서 "우리가 진짜 해결해야 할 문제"를 문장으로 만들 수 있어야 한다는 뜻으로 이해했다.

![정의하기 단계의 핵심 산출물](/images/posts/intel-uiux-special-lecture-day2/image4.jpg)

개발하기 단계에서는 정보구조 설계, 사이트맵, 기능 목록 정리, 유저 플로우, 태스크 플로우, 와이어프레임, 프로토타입, UI 탐색과 스타일 가이드 초안이 나온다. 전달하기 단계에 가면 최종 UI 디자인, 인터랙션 정의, 반응형 설계, 디자인 시스템, 테스트, 사용자 피드백 분석, 반복 설계와 운영 개선이 이어진다. 즉 디자인은 시안 한 장으로 끝나는 일이 아니라 점점 해상도를 높여 가는 과정이었다.

## Step 1. 발견하기: 리서치가 디자인의 출발점

발견하기 단계에서는 리서치가 가장 중요한 시작점으로 소개됐다. 데스크 리서치는 이미 존재하는 자료와 데이터를 활용해 시장과 문제를 간접적으로 이해하는 방식이고, 필드 리서치는 실제 사용자 환경에서 직접 만나거나 관찰하는 방식이다. 여기에 사용자 리뷰 분석이나 설문조사처럼 두 방식 사이를 연결하는 조사도 포함된다.

![발견하기 단계와 Research](/images/posts/intel-uiux-special-lecture-day2/image8.jpg)

![데스크 리서치와 필드 리서치](/images/posts/intel-uiux-special-lecture-day2/image9.jpg)

정량적 리서치와 정성적 리서치를 구분해서 설명한 부분도 기억에 남았다. 정량은 규모와 패턴을 확인하고 가설을 검증할 때 강하고, 정성은 숫자로 설명되지 않는 동기와 감정을 깊게 이해할 때 유용하다. 클릭 수, 전환율, 이탈률, A/B 테스트 같은 지표는 정량에 가깝고, 심층 인터뷰, 관찰 연구, 포커스 그룹, 사용성 테스트는 정성에 가깝다. 둘 중 하나만으로는 부족하고, 서로 보완해야 한다는 점이 핵심이었다.

![정량 리서치와 정성 리서치 비교](/images/posts/intel-uiux-special-lecture-day2/image10.jpg)

경쟁사 분석과 포지셔닝 맵도 소개됐다. 경쟁 서비스를 뜯어보는 건 단순히 따라 하기 위해서가 아니라, 시장 안에서 우리 서비스가 어떤 위치에 설 수 있는지를 판단하기 위한 과정이었다. SWOT 분석도 같은 맥락으로 나왔다. 강점, 약점, 기회, 위협을 분리해서 보면 서비스의 내부 요소와 외부 환경을 동시에 읽을 수 있다.

![경쟁사 분석과 포지셔닝 맵](/images/posts/intel-uiux-special-lecture-day2/image12.jpg)

![SWOT 분석](/images/posts/intel-uiux-special-lecture-day2/image13.jpg)

결국 이 단계에서 중요한 건 "문제를 찾는 것"이지 "바로 해결안을 내는 것"이 아니라는 점이었다. 개발을 하다 보면 뭔가 만들 수 있다는 이유로 솔루션부터 생각하기 쉬운데, 이번 특강은 그 전에 사용자와 맥락을 충분히 이해해야 한다는 점을 계속 강조했다.

## 사용자 리서치는 공감에서 시작한다

사용자 리서치 슬라이드에 나온 "공감이 디자인의 핵심이다"라는 문장이 강하게 남았다. 다른 사람이 보고 느끼고 경험하는 것을 이해하지 못하면 디자인은 무의미한 작업이라는 말이었는데, UX라는 개념을 가장 단순하고 강하게 설명하는 문장처럼 느껴졌다.

![사용자 리서치 개요](/images/posts/intel-uiux-special-lecture-day2/image14.jpg)

설문조사는 다수의 사용자에게 구조화된 질문을 통해 정량적 혹은 정성적 데이터를 수집하는 방법으로 정리됐다. 폐쇄형, 개방형, 리커트 척도 같은 질문 형식이 있고, 질문과 답변 중립성 유지, 한 문항에 한 가지 내용만 묻기, 모호한 보기 피하기 같은 주의사항도 함께 설명됐다. 설문은 쉬워 보이지만 질문을 잘못 쓰면 데이터가 왜곡되기 쉽다는 점이 중요했다.

![설문조사 방법](/images/posts/intel-uiux-special-lecture-day2/image18.jpg)

사용자 인터뷰는 1:1 혹은 그룹 대화를 통해 행동의 이유와 감정을 깊게 탐색하는 정성적 리서치 방법으로 소개됐다. 구조화, 반구조화, 비구조화 인터뷰로 나뉘고, 라포 형성, 비언어적 표현 관찰, 해결책을 먼저 제시하지 않기 같은 원칙이 나왔다. 사용자가 "무엇을 했다"보다 "왜 그렇게 했는가"를 듣는 과정이라는 점에서 상당히 중요해 보였다.

![사용자 인터뷰](/images/posts/intel-uiux-special-lecture-day2/image19.jpg)

사용자 관찰은 실제 환경에서 제품이나 서비스를 사용하는 모습을 직접 보는 방식이다. 인터뷰와 달리 말로 설명되지 않는 행동 패턴을 볼 수 있다는 장점이 있다. 특강에서는 사용자의 행동에 간섭하지 말 것, 해석과 사실을 구분해서 기록할 것, 가능한 자연스러운 상태를 유지할 것 같은 원칙도 함께 정리했다.

![사용자 관찰](/images/posts/intel-uiux-special-lecture-day2/image20.jpg)

## Step 2. 정의하기: 데이터에서 인사이트를 뽑아 문제를 선명하게 만든다

리서치가 끝나면 바로 시안을 그리는 게 아니라, 먼저 데이터를 정리하고 패턴을 찾고 인사이트를 도출한 뒤 문제를 정의해야 한다고 했다. 이때 파인딩과 인사이트를 구분해서 설명한 슬라이드가 좋았다. 파인딩은 리서치 과정에서 발견한 사실 자체이고, 인사이트는 그 사실들 사이의 연관성에서 찾아낸 본질적인 원인이나 새로운 발견이다.

![파인딩과 인사이트의 차이](/images/posts/intel-uiux-special-lecture-day2/image21.jpg)

어피니티 다이어그램은 방대한 양의 아이디어나 데이터를 유사한 속성끼리 그룹화해서 핵심 패턴을 발견하는 방법론으로 소개됐다. 포스트잇을 붙이고 묶고 라벨링하고 상관관계를 보는 식의 전형적인 방식인데, 결국 "흩어진 데이터에서 의미 있는 묶음"을 만드는 과정이라고 이해했다. 아무리 인터뷰를 많이 해도 정리하지 못하면 인사이트로 이어지지 않는다는 점을 잘 보여줬다.

![어피니티 다이어그램 개념](/images/posts/intel-uiux-special-lecture-day2/image18.jpg)

![어피니티 다이어그램으로 인사이트 도출하기](/images/posts/intel-uiux-special-lecture-day2/image19.jpg)

공감맵은 사용자가 말하는 것, 생각하는 것, 행동하는 것, 느끼는 것을 분리해서 보는 도구였다. 같은 사용자를 보더라도 겉으로 표현하는 내용과 실제로 느끼는 감정은 다를 수 있기 때문에, 단순한 요구사항 목록보다 훨씬 풍부하게 사용자 입장을 이해하게 해주는 도구라고 느꼈다.

![공감맵](/images/posts/intel-uiux-special-lecture-day2/image20.jpg)

페르소나는 목표 사용자 집단 안의 대표 유형을 가상의 인물로 구체화한 것이고, 사용자 여정지도는 서비스 이용 전체 과정을 시간 순서대로 시각화한 것이다. 이 둘은 사용자에 대한 해석을 팀 안에서 공유 가능한 형태로 만들기 때문에 기획자, 디자이너, 개발자가 같은 사람을 상상하며 일할 수 있게 해주는 공통 언어처럼 보였다.

![페르소나](/images/posts/intel-uiux-special-lecture-day2/image21.jpg)

![사용자 여정지도](/images/posts/intel-uiux-special-lecture-day2/image22.jpg)

## Step 3. 개발하기: 구조와 흐름을 구체적인 화면으로 바꾸기

개발하기 단계는 말 그대로 코드를 짜는 개발이 아니라, 디자인 솔루션을 발전시키는 단계였다. 아이데이션으로 가능한 해결안을 넓게 발산하고, 정보구조도와 플로우차트로 서비스의 구조와 경로를 정리하고, 와이어프레임으로 화면 배치를 설계하고, 프로토타입으로 실제처럼 동작하게 만드는 흐름이 이어졌다.

![아이데이션](/images/posts/intel-uiux-special-lecture-day2/image24.jpg)

정보구조도는 전체 콘텐츠와 기능을 계층 구조로 시각화한 자료다. 사이트맵이나 메뉴 체계가 여기에 해당한다. 이 구조가 엉켜 있으면 사용자도 길을 잃기 쉽기 때문에, UI를 예쁘게 다듬기 전 구조를 먼저 정리하는 이유가 납득됐다.

![정보구조도](/images/posts/intel-uiux-special-lecture-day2/image25.jpg)

플로우차트는 특정 목적을 달성하기 위해 사용자가 거치는 화면 이동 경로와 의사결정 과정을 도식화한 것이다. 회원가입, 상품 구매, 예약 완료 같은 과정을 설계할 때 유용해 보였다. 특히 예외 처리와 분기 흐름을 미리 보게 해준다는 점에서 개발자 입장에서도 굉장히 도움이 될 것 같았다.

![플로우차트](/images/posts/intel-uiux-special-lecture-day2/image26.jpg)

와이어프레임은 색, 이미지, 폰트보다 레이아웃과 버튼 위치 같은 핵심 기능에 집중한 화면 설계도다. 시각적 디테일에 휘둘리지 않고 구조를 먼저 검증할 수 있다는 장점이 있어 보였다. 프로토타입은 여기에 클릭, 스크롤, 전환 효과 같은 인터랙션을 넣어 실제 서비스처럼 작동하게 만든 동적 모형이다. 기능이 있는 척만 하는 그림이 아니라, 실제로 사용 흐름을 점검할 수 있는 테스트 도구라는 점이 중요했다.

![와이어프레임](/images/posts/intel-uiux-special-lecture-day2/image27.jpg)

![프로토타입](/images/posts/intel-uiux-special-lecture-day2/image29.jpg)

## Step 4. 전달하기: 시스템화하고 검증하고 개선한다

전달하기 단계에서 가장 인상적이었던 것은 디자인 시스템이었다. 강의에서는 디자인 시스템을 서비스에 사용되는 UI 요소의 규칙과 컴포넌트를 정의하고 재사용 가능하게 정리한 디자인 가이드라인과 코드의 집합이라고 설명했다. 버튼, 컬러, 타이포, 간격 규칙이 정리되어 있으면 화면마다 새로 만들지 않아도 되고, 팀 전체의 일관성도 유지된다.

![전달하기 단계](/images/posts/intel-uiux-special-lecture-day2/image28.jpg)

![디자인 시스템](/images/posts/intel-uiux-special-lecture-day2/image30.jpg)

사용성 테스트 사례는 정말 직관적이었다. 장바구니 화면에서 7명 중 1명만 Edit 버튼을 발견했고, 대부분은 리스트를 누르거나 밀어보면 수정할 수 있을 거라고 생각했다는 예시가 나왔다. 디자이너가 "당연히 보이겠지"라고 생각한 요소가 실제 사용자에게는 전혀 다르게 해석될 수 있다는 뜻이다. 이 슬라이드를 보면서 테스트 없이 확신하는 건 위험하다는 걸 실감했다.

![사용성 테스트 사례](/images/posts/intel-uiux-special-lecture-day2/image31.jpg)

A/B 테스트는 두 가지 이상 시안을 사용자에게 무작위로 노출해 어떤 디자인이 클릭이나 구매 같은 목표를 더 잘 달성하는지 비교하는 방법으로 정리됐다. 취향 싸움으로 끝내지 않고 실제 데이터로 판단하는 방식이라는 점에서, 디자인도 결국 검증 가능한 가설 실험이라는 생각이 들었다.

## 들으면서 남긴 내 생각

이번 3월 4일 특강은 "디자인은 감각이 아니라 과정이다"라는 말로 정리할 수 있을 것 같다. 좋은 결과물은 갑자기 떠오른 아이디어 하나에서 나오는 게 아니라, 리서치로 문제를 제대로 보고, 인사이트를 뽑고, 사용자 모델을 만들고, 구조를 설계하고, 검증하면서 다듬는 반복에서 나온다.

특히 개발을 공부하는 입장에서 많이 와닿았던 건, 디자인 산출물 대부분이 사실 개발 전에 굉장히 유용한 정리 도구라는 점이다. IA는 구조를 보여주고, 플로우차트는 분기와 예외를 보여주고, 와이어프레임은 필요한 화면을 보여주고, 프로토타입은 기대 동작을 보여준다. 이런 단계가 잘 잡혀 있으면 개발도 훨씬 덜 흔들릴 수밖에 없다.

또 하나는 사용자 리서치와 테스트의 중요성이었다. 우리는 종종 "사용자가 이렇게 하겠지"라고 쉽게 가정하지만, 실제 사용자 행동은 생각보다 다를 수 있다. 그래서 이번 강의에서 가장 크게 남은 메시지는, 좋은 UI/UX는 추측이 아니라 관찰과 검증 위에서 만들어진다는 점이다.

3월 3일 특강이 UI/UX를 보는 눈을 넓혀 준 시간이었다면, 3월 4일 특강은 그 눈으로 실제 프로젝트를 어떻게 진행해야 하는지 알려준 시간이었다. 앞으로 프로젝트를 할 때도 막연히 화면부터 만드는 대신, 문제 정의와 사용자 흐름, 검증 단계를 더 의식하게 될 것 같다.
