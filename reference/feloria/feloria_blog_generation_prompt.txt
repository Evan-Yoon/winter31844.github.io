Feloria 블로그 초안 작성용 작업 지시

지금부터 너는 “스타일/말투는 이미 학습되어 있고, 내용 정리와 근거 연결만 정확히 하면 되는 상태”라는 전제로 작업한다.
중요한 것은 문체가 아니라 사실 관계, 흐름, 구현 포인트, 설계 의도를 정확하게 복원하는 것이다.

사용 자료
1. manifest.csv
2. commit_map.csv
3. 필요한 feloria_chat_chunk_XX.md 파일들
4. GitHub 커밋 기록 요약(이미 commit_map.csv에 반영됨)

작업 원칙
- 추측하지 말고 자료에 있는 내용만 바탕으로 쓴다.
- 같은 내용을 반복 나열하지 말고, “왜 방향이 바뀌었는지 / 무엇이 구현됐는지 / 어떤 기준으로 구조를 잡았는지”를 중심으로 정리한다.
- 커밋 메시지는 단순 나열하지 말고, 여러 커밋을 하나의 개발 단계로 묶어 서술한다.
- README나 manifest에만 있고 chunk 근거가 없는 내용은 약하게 다루고, chunk와 commit_map 양쪽에서 확인되는 내용은 강하게 다룬다.
- 글은 “기능 목록”이 아니라 “개발 흐름이 보이는 회고형 기술 블로그”로 작성한다.
- 너무 장황한 도입은 줄이고, 프로젝트 배경 → 방향 전환 → 시스템 구현 → 데이터 설계 → 시행착오 → 확장/폴리시 → 회고 순으로 정리한다.
- 코드 전문을 길게 인용하지 말고, 구조와 의도 중심으로 설명한다.
- 실제로 확인되지 않은 성과(예: 완성 출시, 유저 반응, 성능 수치)는 쓰지 않는다.
- 포켓몬/쯔꾸르 감성은 “참고한 방향” 정도로만 다루고, 독창적 구조로 재설계했다는 점을 분명히 한다.

먼저 해야 할 일
1. manifest.csv를 읽고 chunk를 주제별로 분류한다.
2. commit_map.csv를 읽고 개발 단계를 시간 순으로 5~8개 정도의 묶음으로 재구성한다.
3. 각 단계마다 근거로 사용할 chunk를 골라서 메모한다.
4. 그 후에만 블로그 본문을 작성한다.

추천 개발 단계 묶음 예시
- 프로젝트 발상과 첫 기획
- 장르/구조 재정의와 전면 재시작
- Phaser 전환과 기본 씬 구조 구축
- 전투/포획/파티/도감 등 핵심 루프 형성
- 맵/대화/NPC/퀘스트로 실제 RPG화
- 성장/진화/경제/아이템 확장
- 그래픽/이펙트/오디오/자산 정리
- 남은 과제와 회고

출력 형식
반드시 아래 순서로 출력해라.

[1] 자료 해석 메모
- 개발 단계를 5~8개로 묶은 목록
- 각 단계에서 참고한 commit_hash 묶음
- 각 단계에서 참고한 chunk_id 목록
- 각 단계의 한 줄 요약

[2] 블로그 구조안
- 제목 후보 3개
- 섹션 목차
- 각 섹션 핵심 포인트 2~4개

[3] 블로그 본문 초안
아래 구조를 기본으로 작성
1. 프로젝트 개요
2. 왜 이런 게임을 만들게 되었는가
3. 방향이 여러 번 바뀐 이유
4. Phaser 기반으로 다시 구조를 세운 과정
5. 핵심 시스템 구현
   - 전투
   - 포획/상태 유지
   - 파티/도감/성장
6. 월드 구성
   - 맵
   - NPC
   - 대화/퀘스트
7. 데이터 중심 설계
   - 크리처 데이터
   - 스킬/맵/에셋 레지스트리
8. 시각/연출/오디오 폴리시
9. 개발하면서 겪은 문제와 판단
10. 회고

본문 작성 규칙
- commit hash는 본문에 과하게 드러내지 말고, 필요한 경우만 괄호로 짧게 언급한다.
- 각 섹션은 “무엇을 만들었다”보다 “왜 그렇게 구조를 잡았는지”를 먼저 설명한다.
- 구현 범위가 넓더라도 글은 하나의 흐름으로 읽히게 정리한다.
- 자료상 명확한 근거가 있는 시스템만 확정적으로 서술한다.
- 장문의 bullet list보다 자연스러운 문단 중심으로 작성한다.

추가 요청
- 마지막에 “이 프로젝트에서 다음 글로 분리하면 좋은 주제” 5개를 제안해라.
- 마지막에 “블로그 작성에 추가로 보면 좋은 chunk”도 priority 기준으로 5개 추천해라.
