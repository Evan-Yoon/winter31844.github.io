안티그래비티용 프롬프트
당신은 현재 내 프로젝트의 게임 콘텐츠/에셋 연결 담당 개발자입니다.
목표는 이미 개별 크롭 완료된 고양이 포켓몬 100종 스프라이트 PNG를 프로젝트에 체계적으로 연결하고, 현재 구현된 마을에 해당하는 캐릭터만 실제 게임에서 등장하도록 데이터 구조를 정리하는 것입니다.
현재 상황
고양이 포켓몬 스프라이트 PNG 100종은 이미 개별 파일로 정리되어 있음
각 파일명은 캐릭터 key와 1:1 대응되도록 사용 예정
하지만 게임 내 모든 지역/마을이 아직 완성되지 않았으므로, 지금은 현재 구현된 마을에 필요한 캐릭터만 연결
나머지 캐릭터는 추후 맵 확장/패치 시 순차적으로 추가
전설 클래스 캐릭터들은 플레이어가 초반에 직접 획득하거나 일반 필드에서 등장하지 않음
대신 chapter1 마지막에 봉인이 풀릴 때, 전설 클래스 캐릭터들이 잠깐 등장하는 연출용 데이터/연결 구조는 미리 고려

작업 목표
다음 사항을 만족하는 구조와 코드를 제안/작성해줘.
1. 에셋 key와 파일 경로를 정확하게 연결
각 캐릭터 PNG를 정확한 key 값으로 fetch/load 가능하게 만들 것
파일명과 key 값이 다를 가능성을 고려해서, 하드코딩 난립 대신 중앙 집중형 매핑 구조를 만들 것
예:
LEAFKIT -> /assets/creatures/leafkit.png
BRAMBLECAT -> /assets/creatures/bramblecat.png
경로, key naming rule, 대소문자 규칙을 일관되게 정리할 것
이후 캐릭터가 추가되어도 유지보수하기 쉽게 설계할 것
2. 현재 구현된 마을 기준으로만 등장 캐릭터 연결
모든 100종을 한 번에 게임 로직에 투입하지 말 것
현재 만들어진 마을/지역 목록만 기준으로 등장 가능한 캐릭터를 매핑할 것
예:
starter village / forest village / riverside village 등 현재 실제 존재하는 마을만 우선 연결
각 마을마다
등장 가능한 캐릭터 목록
희귀도 또는 출현 비중
기본 속성(예: grass / fire / water / rock / shadow 등)
를 관리할 수 있게 구조화할 것
3. 아직 미구현 지역용 캐릭터는 “보류 상태”로 분리
아직 대응되는 마을/맵이 없는 캐릭터는 삭제하지 말고, reserve / futurePatch / unreleased 같은 상태값으로 관리
즉, 에셋 파일과 기본 데이터는 존재하지만
현재 스폰되지 않음
현재 플레이어가 획득 불가
패치 시 활성화 가능
이런 식으로 확장 가능한 구조로 설계할 것
4. 전설 클래스는 chapter1 엔딩 연출 전용으로 우선 연결
전설 클래스 예시:

대신 chapter1 마지막 봉인 해제 이벤트에서 잠깐 등장 가능한 구조를 마련할 것
예:
availability: "cutscene_only"
unlockPhase: "chapter1_ending"
spawnable: false
추후 chapter2 이상에서 정식 해금될 수 있도록 확장 가능하게 설계할 것
5. 결과물 형태
다음 결과물을 원함:
추천 폴더 구조
캐릭터 메타데이터 구조
마을별 등장 매핑 구조
전설 전용 관리 구조
에셋 key를 로드하는 코드 예시
추후 패치 시 캐릭터를 추가하는 방법 가이드

구현 방향 제약
지금 당장 전체 100종을 다 플레이 가능 상태로 열지 말 것
“현재 구현된 마을만 연결”이 최우선
데이터 구조는 확장성, 유지보수성, 추가 패치 용이성을 최우선으로 설계
key 값, 파일명, 경로가 꼬이지 않도록 단일 소스 오브 트루스(single source of truth) 구조를 만들 것
가능하면 TypeScript 기준으로 작성할 것
상수/맵/enum/interface를 적절히 사용해서 오타를 줄일 것

원하는 출력 스타일
바로 프로젝트에 넣을 수 있는 수준의 실전형 코드
설명은 짧고 명확하게
