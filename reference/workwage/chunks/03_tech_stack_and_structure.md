# Chunk 03 — 기술 선택과 구조 이해

## 이 청크의 역할
이 프로젝트를 어떤 기술로 구현했고, 왜 그 조합을 택했는지 설명한다.

## 핵심 흐름
- 구현 언어와 프레임워크는 **Flutter + Dart**로 확정되었다.
- 초기에는 Python이나 Java로 시작하는 것인지 혼동이 있었지만, 결론은 화면과 로직 모두 Dart 안에서 처리하는 구조였다.
- Java 경험이 Dart 문법 이해에 도움이 된다는 식으로 진입장벽을 낮췄다.
- 상태 관리는 **Provider / ChangeNotifier** 계열로 정리되었고, 로컬 저장은 **SharedPreferences**를 중심으로 구상되었다.
- 공개 저장소의 README와 폴더 구조상 `lib/models`, `lib/providers`, `lib/screens`, `lib/utils`, `lib/widgets`로 분리된 구조를 사용했다.

## 저장소 공개 정보로 확인되는 방향
- 앱 이름은 현재 README 기준으로 **WorkWage**로 표기되어 있다.
- 주요 기능 설명에는 근무 기록 입력, 목표 금액, 프리미엄 기능, 공휴일 자동 반영, PDF/Excel/CSV export, 알림, 구독 기능 등이 포함되어 있다.
- `lib` 폴더가 models/providers/screens/utils/widgets로 분리되어 있어, UI와 계산/상태를 어느 정도 나누려는 의도가 보인다.

## 블로그에 넣으면 좋은 포인트
- Flutter 입문 프로젝트였지만, “화면만 만드는 앱”이 아니라 계산 로직과 상태 관리가 중요한 앱이었다.
- Java와 유사한 문법 덕분에 계산 클래스를 설계하는 데 심리적 장벽이 낮았다.
- 단일 파일에서 끝내기보다, 초반부터 model/provider/screen 분리를 의식한 점을 배운 부분으로 연결할 수 있다.

## 블로그 문장 후보
- “Flutter를 처음 다뤘지만, 급여 계산 로직이 분리된 구조를 만들고 싶어서 모델·상태·화면을 나누는 방향을 초반부터 의식했다.”
- “Dart는 낯설었지만, Java와 비슷한 부분이 많아 계산 클래스를 잡아가는 데 큰 부담은 없었다.”
