# Agent Guide: `project` 카테고리

이 문서는 `category: project` 글을 작성하는 에이전트를 위한 가이드다.
반드시 [style.md](../style.md)의 GLOBAL RULES를 먼저 따르고, 이 문서의 규칙을 추가로 적용한다.

---

## 1. 이 카테고리가 담는 것

`project`는 **실제로 만든 것의 기록**이다. 팀 프로젝트 일별 개발 일지, 캡스톤 결과 정리, 개인 앱 개발 회고 등이 포함된다.
"무엇을 만들었는가"보다 **"왜 그렇게 결정했는가, 어디서 막혔는가, 어떻게 해결했는가"**가 핵심이다.

---

## 2. 서브카테고리별 성격

### `team-project` — 팀 프로젝트 일별 개발 일지

- 다인 팀으로 진행하는 프로젝트를 날짜 단위로 기록하는 시리즈다.
- 타이틀 형식: `[프로젝트명] Day N 소제목`
- 예: `[Walkmate] Day 1 시각장애인 AI 보행 앱 시작`
- 당일 목표와 달성 결과를 표로 먼저 보여주는 것이 효과적이다.
- Git commit hash 및 커밋 메시지를 직접 인용하면 실제감이 살아난다.
- 기술적 결정(왜 이 스택/모델을 골랐는가)을 이슈 → 솔루션 구조로 설명한다.

### `personal-project` — 개인 프로젝트 회고 / 개발 기록

- 혼자 만드는 앱·서비스의 개발 과정이나 회고 글이다.
- 여러 파트로 나눠 시리즈로 쓰는 경우와, 최종 회고 한 편으로 정리하는 경우 모두 존재한다.
- 아이디어 선택 이유 → 기술 결정 과정 → 실제로 막힌 문제 → 해결 방식 → 결과 및 느낀 점 흐름을 유지한다.

### `capstone` (또는 `team-project` 내 캡스톤) — 발표 중심 프로젝트 정리

- 수업 캡스톤 또는 해커톤 결과물을 정리하는 글이다.
- 프로젝트 과제의 요구사항, 실제 작업 흐름, 발표 결과까지 한 편에 담는다.
- 데이터 탐색 → 전처리 → 모델 비교 → 결과 해석 → 제안까지 서술형으로 적는다.

---

## 3. 문체 및 구조

- 개발 글이지만 **기술 문서가 아니라 이야기**처럼 써야 한다.
- 기술적 선택의 이유를 반드시 포함한다. "Flutter를 썼다"가 아니라 "왜 Flutter를 골랐는가".
- 실패하거나 돌아간 경험이 있으면 반드시 포함한다. 그것이 가장 읽힐 만한 부분이다.
- 코드 블록을 사용할 경우 핵심 로직만, 그리고 왜 이 코드가 중요한지 한 줄 이상 설명한다.
- 인용 블록(`>`)은 프로젝트의 핵심 철학이나 인상 깊은 문장을 강조할 때 사용한다.

### 권장 섹션 순서 (team-project 일지)

1. **인용 블록** — 프로젝트의 핵심 가치나 철학 한 줄 (선택)
2. **Today's Mission & Results** — 당일 목표와 완료 여부 표
3. **GitHub Commit History** — 커밋 해시와 변경 내용 (선택)
4. **기술적 이슈와 해결** — Issue → Solution 구조
5. **Daily Report Summary** — 소요 시간, 핵심 결과물, 팀 작업 내용
6. **Reflections** — 느낀 점, 시각이 바뀐 순간
7. **Next Plan** — 다음 날 목표

### 권장 섹션 순서 (personal-project 회고)

1. **이 프로젝트를 선택한 이유** — 아이디어 출발점
2. **핵심 결정과 그 이유** — 기술 스택, 구조, 방향 전환 등
3. **실제로 어려웠던 것** — 기술적 난관, 경계값 처리, 예상치 못한 이슈
4. **계산기에서 제품으로** (변곡점) — 규모나 성격이 바뀌는 순간
5. **느낀 점** — 프로젝트 전체를 통해 얻은 것
6. **배운 점** — 기술적 + 비기술적 배움
7. **다음에 다듬고 싶은 것** — 솔직한 부족한 점

## 4. Frontmatter 작성법

```yaml
---
title: |
  [Walkmate] Day 1
  시각장애인 AI 보행 앱 시작
slug: walkmate-day1-ai-setup
date: 2026-02-09
author: Evan Yoon
category: project
subcategory: team-project      # team-project | personal-project
description: "한 문장 요약. 기술 스택이나 핵심 결정을 담으면 좋다."
thumbnail: /images/posts/walkmate/cover.png
tags:
  - 프로젝트명-소문자
  - 기술-스택
  - team-project 또는 personal-project
readTime: 8
series: "Walkmate"             # 시리즈 이름
seriesOrder: 1
featured: true                 # 메인 노출 원하는 경우 true
draft: false
toc: true
---
```

**주의사항**
- `title`이 두 줄일 경우 yaml multi-line(`|`) 사용.
- `thumbnail`은 `/images/posts/[프로젝트명-소문자]/cover.png` 경로 패턴.
- `featured: true`는 팀 프로젝트 시리즈 첫 편, 또는 개인 프로젝트 완성 회고에 사용.
- `tags`에는 반드시 프로젝트명 태그 포함 (예: `walkmate`, `workwage`, `mongle`).

## 5. 실제 글 패턴 예시

### team-project 일지 — 기술 결정 서술 패턴

```markdown
### ⚠️ Issue Situation

기존의 서버 통신 방식(API)은 지연 시간(Latency) 문제로 인해 실시간 보행 보조에
한계가 있었다. 보행 중 1~2초의 지연은 치명적인 사고로 이어질 수 있기 때문이다.

### 💡 Solution: On-device AI

1. **Model Selection**: 최신 YOLOv11 Nano 모델을 선택하여 연산량을 최소화했다.
2. **Quantization**: 양자화 적용 시 모델 용량을 6MB 이하로 줄일 수 있음을 확인했다.
```

→ 이슈를 먼저 제시하고 이유를 설명한 뒤 해결 방식을 서술한다.

### personal-project 회고 — 전환점 서술 패턴

```markdown
처음에는 계산기였다. 기록이 붙으면서 기록 관리 앱이 됐다. 통계와 export가 붙으면서
리포트 도구가 됐다. 백업과 구독이 붙으면서 서비스 형태를 갖추기 시작했다.
```

→ 제품의 성격 변화를 시간 흐름으로 압축해 서술한다.

### 기술 결정에 이유를 붙이는 패턴

```markdown
Flutter를 택한 이유는 명확했다. 화면과 계산 로직을 Dart 하나로 처리할 수 있다는 것.
모바일 앱에서 별도 백엔드 언어를 같이 배우는 것은 7일이라는 기간 안에 맞지 않았다.
```

→ 기술 선택 = 이유 + 대안 비교 + 제약 조건.

## 6. 하면 안 되는 것

- "오늘 X를 개발했습니다. 다음에는 Y를 할 예정입니다."처럼 업무 보고서처럼 쓰기
- 기술 스택 목록만 나열하고 이유 없이 끝내기
- 막힌 부분을 생략하거나 "쉽게 해결됐다"로 넘기기
- 팀 프로젝트인데 "나"만 등장하거나, 팀원 이름이 전혀 안 나오기
- 완성된 것만 자랑하고 시행착오를 지우기
